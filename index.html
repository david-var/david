Migrating to Apollo iOS 2.0: What You Need to Know

December 2025

My team recently completed the migration from Apollo iOS 1.x to 2.0.3. This is a breaking change that requires careful planning—here's what we learned and what to watch out for if you're considering the upgrade.

Why migrate now?

  • Swift 6 & async/await — Apollo 2.0 replaces callback-based APIs with structured concurrency. If you're adopting Swift 6, this alignment is essential
  • Sendable conformance — Most types now conform to Sendable, eliminating common thread-safety issues in concurrent code
  • CocoaPods is gone — 2.0 drops CocoaPods support entirely. If you're still on 1.x, you're falling behind on security patches
  • Minimum iOS 15+ — The new deployment target may require dropping support for older devices

Key migration steps:

  • Re-run code generation — Update your dependency and run codegen with the 2.0 engine first. This surfaces most breaking changes immediately
  • Refactor custom interceptors — The old ApolloInterceptor protocol is split into discrete types: GraphQLInterceptor, HTTPInterceptor, CacheInterceptor, and ResponseParsingInterceptor. We had existing interceptors for auth tokens and custom headers—these needed to be refactored to the new HTTPInterceptor protocol with async/await signatures
  • Update GraphQL response handling — GraphQLResult is now GraphQLResponse and is generic over the operation type, not just the data
  • Handle cache misses differently — Cache misses now return nil instead of throwing errors. Update any error-handling logic accordingly
  • SPI imports for custom scalars — If you have custom scalar types, you'll see "does not conform to protocol" errors. Fix these by updating the import to @_spi(Execution) @_spi(Internal) import ApolloAPI. Apollo now uses SPI attributes to protect internal APIs

The full migration guide is available in the Apollo iOS documentation.
